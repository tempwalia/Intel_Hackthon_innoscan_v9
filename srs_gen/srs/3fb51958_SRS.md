# Software Requirements Specification (SRS)  
# Smart Energy Consumption Optimizer  

## 1. PROJECT OVERVIEW  
- **Title**: Smart Energy Consumption Optimizer  
- **Description**: Predict and reduce office energy usage using IoT sensor data.  
- **Problem Statement**: Facilities teams lack proactive insights to reduce peak energy costs.  
- **Expected Outcome**: Dashboard with consumption forecasts, anomaly alerts, and optimization recommendations.  
- **Programming Language**: Python  
- **Approach/Solution**: Ingest sensor streams, train forecasting models, and trigger rule-based optimization suggestions.  
- **Technology Stack**: Python, Pandas, Prophet, FastAPI, Streamlit  
- **Project Complexity**: Medium  
- **Team Size**: 1 developer(s)  
- **Project Manager**: shyaam123  
- **Timeline**: 10-15 days  

---

## 2. FUNCTIONAL REQUIREMENTS  
### Input/Processing  
- Ingest real-time IoT sensor data (temperature, motion, power usage).  
- Preprocess data using Pandas for normalization and timestamp alignment.  
- Validate data integrity (e.g., missing values, outliers).  

### Core Engine  
- Train forecasting models using Prophet for short-term energy consumption prediction.  
- Detect anomalies (e.g., sudden spikes in power usage) via threshold-based rules.  
- Generate optimization recommendations (e.g., adjust HVAC settings, schedule equipment maintenance).  

### API Endpoints  
- **POST /ingest**: Upload sensor data streams.  
- **GET /forecast**: Retrieve energy consumption forecasts (hourly/daily).  
- **POST /alert**: Trigger anomaly alerts (e.g., high usage, equipment failure).  
- **GET /recommendations**: Fetch optimization suggestions.  

### Output Format  
- Dashboard visualizations (line charts, heatmaps) via Streamlit.  
- JSON responses for API endpoints (e.g., forecasts, alerts).

## 4. TECHNICAL ARCHITECTURE  
### Stack  
- **Backend**: FastAPI (Python 3.9+) for RESTful APIs.  
- **Data Processing**: Pandas (v1.5.3) for data manipulation.  
- **Modeling**: Prophet (v1.1.1) for time-series forecasting.  
- **Frontend**: Streamlit (v1.24.0) for interactive dashboards.  

### Components  
- **Data Ingestion Module**: MQTT/HTTP endpoints for sensor data.  
- **Model Training Pipeline**: Scheduled Prophet model retraining.  
- **Dashboard**: Real-time visualization of forecasts and alerts.  

### Model/Framework Details  
- **Prophet**: Seasonality and trend analysis for energy patterns.  
- **Rule-Based Logic**: Thresholds for anomaly detection (e.g., >95% usage threshold).

## 5. API SPECIFICATION  
### Request/Response Examples  
#### POST /ingest  
**Request**:  
```json
{
  "sensor_id": "sensor_001",
  "timestamp": "2023-10-05T14:30:00Z",
  "power_usage": 120.5,
  "temperature": 22.3
}
```  
**Response**:  
```json
{
  "status": "success",
  "message": "Data ingested",
  "timestamp": "2023-10-05T14:30:00Z"
}
```  

#### GET /forecast  
**Request**:  
```json
{
  "start_time": "2023-10-06T00:00:00Z",
  "end_time": "2023-10-06T23:59:59Z"
}
```  
**Response**:  
```json
{
  "forecast": [
    {"timestamp": "2023-10-06T00:00:00Z", "predicted_usage": 115.2},
    {"timestamp": "2023-10-06T12:00:00Z", "predicted_usage": 130.8}
  ]
}
```

## 6. DATA & CATEGORIES  
### Predefined Classifications  
- **Sensor Types**: Temperature, Motion, Power Usage.  
- **Data Types**:  
  - `sensor_id`: String (max 20 chars).  
  - `timestamp`: ISO 8601 format.  
  - `power_usage`: Float (unit: kWh).  
- **Anomaly Categories**:  
  - High Usage (threshold >95% of baseline).  
  - Equipment Failure (sudden drop in sensor values).  

---

## 7. INPUT CONSTRAINTS  
- **Field Limits**:  
  - `sensor_id`: Max 20 characters.  
  - `power_usage`: Float between 0 and 1000.  
- **Format Requirements**:  
  - Timestamp: ISO 8601 (e.g., "2023-10-05T14:30:00Z").  
  - JSON payload for all API requests.
